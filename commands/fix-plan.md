Fix 수정계획 문서를 생성합니다.

인수 형식: `문제 설명` (예: `/fix-plan 저장 후 데이터가 사라지는 버그`)

**아래 단계를 순서대로 실행하세요:**

## 1단계: 인수 및 날짜 확인

`$ARGUMENTS`가 비어 있으면 어떤 문제인지 사용자에게 물어보고 중단하세요.

오늘 날짜를 YYMMDD 형식으로 확인하세요:
```bash
date +%y%m%d
```

파일명 핵심단어는 `$ARGUMENTS`를 2~5글자로 압축합니다 (예: 저장버그, 감지중단, 로그오류).

## 2단계: 프로젝트 설계 원칙 파악

프로젝트의 CLAUDE.md 파일을 모두 찾아 읽으세요:

```bash
find . -name "CLAUDE.md" -not -path "./.git/*" 2>/dev/null
```

찾은 순서대로 읽으세요. 없으면 이 단계를 건너뜁니다.

## 3단계: 관련 코드 탐색

`$ARGUMENTS`를 바탕으로 관련 파일을 읽고 원인을 분석하세요.

- Grep으로 관련 키워드 검색
- 의심되는 파일 Read로 확인
- 원인 후보를 1순위, 2순위 순으로 정리

## 4단계: 수정계획 문서 작성

`Fix/YYMMDD_핵심단어_수정계획.md` 파일을 아래 구조로 작성하세요.

`Fix/` 폴더가 없으면 생성합니다. 같은 날짜 파일이 이미 있으면 다른 핵심단어를 사용하세요.

```
- 사용자가 목격한 증상

- 파일: `경로/파일.py` L번호

## 수정 대상 파일

## Phase 실행 프로토콜

> **이 계획서를 열고 Phase를 구현하는 Claude는 아래 프로토콜을 반드시 따른다.**
>
> 1. 각 Phase의 모든 코드 수정 및 커밋이 완료되면
> 2. **eval-plan 서브에이전트를 포그라운드로 즉시 실행한다** — `evaluate` 모드, 커밋 범위는 해당 Phase의 커밋 수만큼 (`HEAD~N`)
> 3. 먹통 관련 수정(감지 루프, 스레드, 타이머, 상태 관리)이 포함된 경우 **eval-freeze 서브에이전트도 함께 실행한다**
> 4. 평가 결과를 사용자에게 보고한 후 다음 Phase 진행 여부를 확인한다
> 5. Critical/High 발견 시 — 다음 Phase 진행 전에 수정 여부를 사용자에게 확인한다
> 6. 각 작업 항목 완료 즉시 체크박스를 `[x]`로 업데이트한다 — 중단 후 재개 시 진행 상태 파악용

## 단계별 수정 계획

> 각 Phase는 새로운 대화세션에서 시작한다.
> 중단되어도 다음 세션에서 이 파일을 열어 이어서 진행 가능.

### Phase 1 — [작업 제목]

- [ ] 작업 항목
- [ ] 커밋: `메시지`
- [ ] eval-plan evaluate 자동 실행

### Phase 2 — [작업 제목] (해당 시)
- [ ] 작업 항목
- [ ] 커밋: `메시지`
- [ ] eval-plan evaluate 자동 실행

### Phase N — 검증
- [ ] 현장 테스트
- [ ] 로그 확인

## 참고

- 관련 이력: (과거 유사 버그 있으면 기록)
```

## 5단계: 사전 검토 자동 실행

계획서 생성이 완료되면 **즉시 eval-plan 서브에이전트를 실행**하여 사전 검토를 수행하세요.

서브에이전트에게 전달할 내용:
- 모드: `review`
- 검토 대상: 방금 생성한 계획서 파일 경로
- 지시: `eval-plan` 에이전트 정의에 따라 review 모드로 실행

서브에이전트 실행은 백그라운드가 아닌 **포그라운드**로 실행하여 결과를 기다리세요.

## 6단계: review 반영 여부 확인

review 이슈 목록을 분석하여 **각 이슈별로 계획서 반영 여부를 사용자에게 질문**하세요.

질문 형식:
- 이슈마다 한 줄로 요약 + 내 추천 옵션을 명시
- 추천 옵션에는 `← 추천` 표기

예시:
```
review 결과를 계획서에 반영하기 전에 확인합니다.

1. [High-1] faulthandler 파일 핸들을 모듈 변수에 저장 (GC 문제 방지)
   → A) 반영한다 ← 추천  B) 반영 안 한다

2. [Medium-1] Phase 1 체크리스트에 L145, L149 변경 항목 명시
   → A) 반영한다 ← 추천  B) 반영 안 한다

3. [Low-2] _open_settings() 추가 try-except 항목 제거 (이미 구현됨)
   → A) 반영한다 ← 추천  B) 반영 안 한다

4. 추가 관찰 — Bug B race condition 내용을 Phase 2에 추가
   → A) 반영한다 ← 추천  B) 반영 안 한다

각 항목 번호에 A 또는 B로 답해주세요. (예: 1A 2A 3B 4A)
```

사용자 응답을 받은 후, 반영 결정된 항목만 계획서에 수정·추가하세요.

## 7단계: 결과 보고

계획서 수정 완료 후 보고하세요:

- 생성된 계획서 경로
- 검토 결과 문서 경로 및 심각도별 건수 요약
- 반영된 항목 / 미반영 항목 구분하여 요약
- Phase 1을 바로 시작할지 확인

구현 완료 후에는 `@eval-plan evaluate`(일반 수정) 또는 `@eval-freeze evaluate`(먹통 관련 수정)로 사후 평가를 수행할 수 있습니다.
