# 프로젝트 커스텀 fix-plan 템플릿
#
# 사용법:
#   이 파일을 프로젝트의 .claude/commands/fix-plan.md 로 복사한 후
#   [PROJECT_NAME], [DESIGN_DOCS] 등 대괄호 항목을 실제 값으로 교체하세요.
#
# 전역 버전(~/.claude/commands/fix-plan.md)과 차이점:
#   - 이 프로젝트 전용 설계 원칙 파일을 명시적으로 지정
#   - 프로젝트 고유 체크 항목 추가 가능
#   - eval-plan 대신 프로젝트 전용 평가 에이전트 지정 가능
# -------------------------------------------------------------------

Fix 수정계획 문서를 생성합니다. ([PROJECT_NAME] 전용)

인수 형식: `문제 설명` (예: `/fix-plan 저장 후 데이터가 사라지는 버그`)

**아래 단계를 순서대로 실행하세요:**

## 1단계: 인수 및 날짜 확인

`$ARGUMENTS`가 비어 있으면 어떤 문제인지 사용자에게 물어보고 중단하세요.

오늘 날짜를 YYMMDD 형식으로 확인하세요:
```bash
date +%y%m%d
```

파일명 핵심단어는 `$ARGUMENTS`를 2~5글자로 압축합니다.

## 2단계: 프로젝트 설계 원칙 파악

아래 문서를 읽으세요 (이 원칙 위반은 Critical로 간주):
- `CLAUDE.md` (루트)
# [DESIGN_DOCS: 추가 설계 원칙 파일 경로를 여기에 나열]
# 예: - `src/core/CLAUDE.md`
# 예: - `src/ui/CLAUDE.md`

## 3단계: 관련 코드 탐색

`$ARGUMENTS`를 바탕으로 관련 파일을 읽고 원인을 분석하세요.

- Grep으로 관련 키워드 검색
- 의심되는 파일 Read로 확인
- 원인 후보를 1순위, 2순위 순으로 정리

## 4단계: 수정계획 문서 작성

`Fix/YYMMDD_핵심단어_수정계획.md` 파일을 아래 구조로 작성하세요.

`Fix/` 폴더가 없으면 생성합니다. 같은 날짜 파일이 이미 있으면 다른 핵심단어를 사용하세요.

---

문서 구조:

```
# [문제 제목] 수정계획

## 문제 현상
- 어떤 상황에서 발생하는지
- 재현 조건 (항상 / 간헐적 / 특정 조건)
- 사용자가 목격한 증상

## 원인 분석

### 1순위: [원인명]
- 파일: `경로/파일` L번호
- 설명: ...

### 2순위: [원인명] (있을 경우)
- ...

## 수정 대상 파일

| 파일 | 수정 위치 | 내용 |
|------|-----------|------|
| `파일명` | `함수명()` | 설명 |

## Phase 실행 프로토콜

> **이 계획서를 열고 Phase를 구현하는 Claude는 아래 프로토콜을 반드시 따른다.**
>
> 1. 각 Phase의 모든 코드 수정 및 커밋이 완료되면
> 2. **eval-plan 서브에이전트를 포그라운드로 즉시 실행한다** — `evaluate` 모드, 커밋 범위는 해당 Phase의 커밋 수만큼 (`HEAD~N`)
> 3. 평가 결과를 사용자에게 보고한 후 다음 Phase 진행 여부를 확인한다
> 4. Critical/High 발견 시 — 다음 Phase 진행 전에 수정 여부를 사용자에게 확인한다
> 5. 각 작업 항목 완료 즉시 체크박스를 `[x]`로 업데이트한다

## 단계별 수정 계획

### Phase 1 — [작업 제목]
- [ ] 작업 항목
- [ ] 커밋: `메시지`
- [ ] eval-plan evaluate 자동 실행

### Phase N — 검증
- [ ] 현장 테스트

## 참고
- 관련 설계 원칙: (CLAUDE.md 해당 섹션)
- 관련 이력: (과거 유사 버그)
```

---

## 5단계: 사전 검토 자동 실행

계획서 생성 완료 후 **즉시 eval-plan 서브에이전트를 포그라운드로 실행**하세요.
모드: `review`, 대상: 방금 생성한 계획서 경로

## 6단계: 결과 보고

- 생성된 계획서 경로
- 검토 결과 문서 경로 및 심각도별 건수 요약
- Critical/High 발견 시: 계획서 수정 필요 여부 사용자에게 확인
